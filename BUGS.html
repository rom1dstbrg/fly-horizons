1
Paiements perso via admin jamais enregistrés en BDD après Stripe
app/api/stripe/webhook/route.ts
▼
Problème
Le webhook Stripe filtre .eq("statut", "en_attente") pour les reservation_perso. Or, les réservations créées via l'admin avec envoyer_paiement = true sont créées avec statut = "payment_pending", pas "en_attente". Le webhook ne les trouve donc jamais et ne met jamais à jour leur statut.
Impact
Le client paie, Stripe confirme, mais la réservation reste bloquée à payment_pending indéfiniment. Aucun email de confirmation envoyé. Romain ne voit pas le paiement dans l'admin.
Correction
Remplacer .eq("statut", "en_attente") par .in("statut", ["en_attente", "payment_pending"]) dans le bloc reservation_perso du webhook.
Critique
Corrigé ✓
2
Emails de confirmation envoyés même si la mise à jour BDD a échoué
lib/actions/reservations.ts — 6 fonctions
▼
Problème
updateStatutReservation, updateStatutReservationPerso, updateReservationDateHeure, updateReservationFields, updateReservationPersoFields, updateReservationRoute — aucune de ces fonctions ne vérifie le résultat de la requête Supabase. Si l'update échoue (RLS, réseau, contrainte), la fonction retourne { success: true } et envoie quand même l'email.
Impact
Client reçoit un email de confirmation de statut alors que la BDD n'a pas changé. Données incohérentes entre ce que voit le client et la réalité.
Correction
Ajouter const { error } = await supabase.from(...).update(...); et retourner une erreur si error est présent, avant d'envoyer l'email.

Voucher marqué "used" avant paiement lors d'une création admin
lib/actions/reservations.ts — createAdminReservation
▼
Problème
Dans createAdminReservation, si un voucher est fourni, il est immédiatement marqué used à la création de la réservation — avant même que le client ait payé. Le flux admin vol sur mesure (createAdminVolMesure) le marque correctement reserved et attend le webhook Stripe. Incohérence entre les deux.
Impact
Si la réservation est annulée avant paiement, le voucher reste used — perdu sans rollback. Le client ne peut plus l'utiliser.
Correction
Si envoyer_paiement = true, marquer le voucher reserved (et laisser le webhook le passer à used). Sinon, marquer directement used.

4
respondToRouteProposal est une action publique avec effets financiers
lib/actions/reservation-edit.ts
▼
Problème
La server action respondToRouteProposal n'a pas de checkAdmin(). Elle est entièrement publique — la seule protection est le token de proposition (UUID). Quand le client accepte, elle change le statut de la réservation ET envoie un lien de paiement Stripe.
Impact
Un token exposé (email forwardé, URL copiée) permet à n'importe qui d'accepter la route à la place du client, de déclencher l'envoi d'un lien de paiement et de modifier l'état de la réservation. C'est voulu pour le client, mais sans contrainte de token à usage unique.
Correction
Ajouter une vérification que le statut de la proposition est encore "pending" avant d'exécuter (déjà partiellement géré via alreadyResponded côté UI, mais pas enforced côté serveur). Idéalement, invalider le token après première utilisation.

7
Rate limiter inopérant en production serverless
lib/rate-limit.ts
▼
Problème
Le rate limiter utilise un Map en mémoire Node.js. Sur Netlify (fonctions serverless), chaque requête peut être traitée par une instance séparée. Chaque instance a son propre Map vide → aucune limite n'est jamais atteinte.
Impact
Les endpoints /api/reservation/submit, /api/vol-sur-mesure/checkout et /contact sont sans protection contre le spam/abuse en production.
Correction
Utiliser Upstash Redis (compatible Netlify Edge) avec @upstash/ratelimit, ou désactiver le rate limiter et s'appuyer sur la protection Netlify/Cloudflare.
Important

16
Envoi newsletter séquentiel — timeout probable dès ~50 abonnés
lib/actions/newsletter.ts
▼
Problème
Les emails sont envoyés dans une boucle for...of avec await séquentiel. À ~100ms par email Resend, 50 abonnés = 5s, 100 abonnés = 10s, ce qui dépasse le timeout d'une server action Next.js.
Impact
La newsletter s'arrête en cours d'envoi, une partie des abonnés ne reçoit rien, aucune erreur visible dans l'admin.
Correction
Utiliser l'API Resend Batch (resend.batch.send([])) qui envoie tout en une seule requête.
